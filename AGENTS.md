## 阅读顺序

后续 agent 进入本项目时，请按以下顺序阅读：

1. `README.md`：了解项目定位、目录结构、运行方式和当前状态。
2. `AGENTS.md`：确认本项目的协作规则、回复偏好、后端接口规则和数据库规则。
3. `docs/api.md`：涉及接口变更时阅读。
4. `docs/troubleshooting.md`：遇到页面渲染、接口 500、路由匹配、前后端联调异常时阅读。
5. `docs/performance.md`：遇到接口变慢、数据库连接抖动或性能基线验证时阅读。
6. `database/schema.sql`：涉及数据库、表结构、字段或示例数据变更时阅读。
7. `.agents/`：如存在更细分的提示词、规范或任务资料，再按文件名和上下文选择阅读。

## 协作规则

- 优先保持文档清晰、短小、可维护。
- 修改项目说明、目录结构或长期规则时，同步更新 `README.md` 或本文件。
- 不要删除用户已有内容，除非用户明确要求。
- 遇到规则冲突时，优先级为：用户当前明确指令 > 本文件规则 > 其他项目文档。
- 用户要求“推送到远程”或“推送到 GitHub”时，默认使用当前已配置的 `origin` 远端和当前分支进行提交与推送。
- 推送前应检查 `git status --short --branch`、远端地址和未跟踪文件，确认 `.env`、依赖目录、运行日志和构建产物没有进入提交范围。
- 如果工作区只有当前项目相关变更，可直接 `git add -A`、提交并 `git push -u origin <branch>`；如果存在明显不相关变更，应先向用户确认提交范围。
- GitHub 认证优先使用本机已配置的 SSH key；认证失败时，再提示用户添加 SSH 公钥或提供临时 PAT。
- 首次推送空仓库时，默认使用 `main` 分支，除非用户指定其它分支。

## 调试记录要求

- 遇到页面空白、详情字段显示“未记录”、接口 500 等问题时，先查看 `docs/troubleshooting.md`。
- 排查前后端联调问题时，优先直接请求后端接口确认真实响应结构，再检查路由匹配顺序和前端字段绑定。
- 修复过的典型问题应补充到 `docs/troubleshooting.md`，避免后续重复踩坑。

## 文件约定

- `README.md` 用于人类和 agent 快速理解项目。
- `AGENTS.md` 用于记录 agent 需要长期遵守的协作规则。
- `docs/api.md` 用于记录接口说明。
- `docs/troubleshooting.md` 用于记录常见联调、路由、接口和页面渲染问题。
- `database/schema.sql` 用于维护 MySQL 建表脚本和示例数据。
- `.agents/` 用于存放更细分的 agent 资料，例如提示词、流程说明、检查清单或任务上下文。

## 后端分层规则

- Controller 层只负责逐项接收请求参数、基础参数校验、鉴权、调用 Logic 和返回统一响应。
- Controller 层不得承载复杂业务判断、数据库查询拼装、事务细节或跨模型业务流程。
- 多个业务 Controller 共享的响应、异常捕获、参数读取和基础类型转换能力，应抽到 Controller 父类。
- Logic 层与 Model 层保持同级目录，不放在 Model 目录内部。
- Logic 层负责具体业务规则、业务状态判断、业务流程编排和事务边界控制。
- Logic 层不得直接拼接数据库查询，不得绕过 Model 层访问数据表。
- Model 层负责数据表字段常量、基础查询、写入、更新、删除、计数和业务可复用数据访问方法。
- Model 层字段名优先使用常量定义，避免业务代码散落字符串字段名。
- Logic 层访问数据必须通过 Model 层已定义的方法完成。
- 业务判断失败时，Logic 层应抛出业务异常，不使用 `null`、`false`、空数组或特殊 ID 表示失败。
- 事务只包裹必要的读写步骤，避免把参数校验、文件处理、响应组装或无关查询放入事务。

## 后端接口规则

- 接口保持前后端分离，响应统一使用 JSON。
- Controller 层每个接口参数必须逐项接收，不直接把完整请求参数包透传给 Logic。
- GET 接口参数通过 query string 接收。
- POST 接口参数通过 body 接收。
- 项目接口不默认使用 RESTful 路径参数风格。
- 写操作统一优先使用 POST，除非项目另有明确接口规范。
- 接口路由应保持明确、稳定、可预测，避免依赖模糊匹配或路由顺序兜底。
- 公开接口和管理接口应在路由、命名和鉴权边界上清晰区分。
- 接口文档变更必须同步更新前端请求代码、后端路由、测试用例和联调说明。

## 参数校验规则

- Controller 层负责基础参数校验，包括必填、基础类型转换、字符串去空格、整数默认值和简单范围判断。
- 必填参数缺失或基础格式不合法时，应抛出统一业务异常或返回统一错误码。
- 重复参数校验逻辑应沉淀到 Controller 父类或可复用校验工具中。
- Logic 层仍需保留业务有效性校验，例如资源是否存在、状态是否允许、关联关系是否满足。
- 参数校验错误文案不得散落在 Controller 或 Logic 中，应通过统一错误码映射获取。

## 响应与错误码规则

- 所有接口响应结构固定为 `{ code, message, data }`。
- `code = 0` 表示成功响应。
- `code != 0` 表示业务错误或接口错误。
- 成功响应的 `data` 返回真实业务数据。
- 错误响应必须保留 `data` 字段，默认返回空数组。
- 错误码应集中定义在统一文件中，错误文案通过错误码映射数组获取。
- 错误码应按模块或错误类型分段管理，避免后期扩展混乱。
- Controller 层捕获业务异常后，应使用异常中的错误码和错误文案返回统一响应。
- 未知错误码必须有兜底文案，避免返回空 message。

## 管理接口与上传安全规则

- 管理接口写操作必须有明确鉴权边界。
- MVP 阶段可使用固定管理令牌，但令牌必须来自环境配置，不写死在代码中。
- 管理鉴权逻辑可先保留在管理 Controller 内；管理接口增多后再抽取管理父类或中间件。
- 上传接口必须校验文件是否存在、文件大小、扩展名和 MIME 类型。
- 上传失败必须返回统一错误码和统一响应结构。
- 上传文件的公开访问 URL 与物理存储路径应保持清晰边界。
- 不可信的客户端参数不得直接参与文件路径拼接。

## 接口性能规则

- 列表接口只返回列表页必要字段，详情接口再返回完整字段。
- 列表接口不得因为展示需要加载大字段、长文本或详情专用字段。
- 需要返回互动状态、统计状态或关联状态时，应使用批量查询，禁止循环内查库。
- 查询条件、排序字段和分页字段必须可解释、可控制。
- 高频公开接口应建立本地性能基线，并记录慢接口排查方法。
- 性能问题优先从数据库连接、SQL 执行、业务逻辑和响应数据量四类方向定位。
- 数据库连接配置应支持连接超时等基础可观测和稳定性配置。
- 不提前引入缓存；只有当数据量、访问量或稳定性问题明确出现时，再评估缓存、对象存储或 CDN。

## 测试与文档规则

- 分层规则、响应结构、错误码、路由风格和数据库约束都应有自动化测试覆盖。
- Controller 结构测试应确保业务 Controller 继承公共父类，并避免直接调用底层响应 helper。
- Logic 分层测试应确保 Logic 不绕过 Model 层直接访问数据库查询 API。
- 参数校验测试应覆盖必填缺失、类型不合法和错误响应 `data` 为空数组。
- 性能基线不应强依赖单元测试耗时，可通过脚本或文档记录稳定环境下的验证方式。
- 修复过的典型联调问题必须沉淀到排查文档中。
- 长期规则变更后，应同步检查 README、接口文档、数据库脚本、测试和排查文档是否需要更新。

## 数据库与表结构规则

- 数据库使用 MySQL，建表脚本应统一维护在项目约定的数据库脚本目录中。
- 数据库字符集统一使用 `utf8mb4`，排序规则统一使用 `utf8mb4_unicode_ci`。
- 数据表统一使用 `InnoDB` 引擎。
- 每张数据表必须包含清晰的表级注释。
- 每个字段必须包含清晰的字段注释，说明字段含义、取值规则或业务约束。
- 时间字段统一使用 Unix 秒级时间戳类型，字段定义为 `INT(10) NOT NULL DEFAULT 0`。
- 业务时间字段不使用 `DATE` 或 `DATETIME` 类型。
- 状态、类型、开关类字段统一使用 `TINYINT(1)`。
- 状态、类型、开关类字段不使用 `ENUM` 类型。
- 搜索索引应优先使用兼容性更好的普通索引或应用层组合查询。
- 不默认使用 `FULLTEXT` 索引；如确需使用，必须先确认目标 MySQL 版本、表引擎和部署环境均支持。
- 需要防重复写入的业务关系，应通过数据库唯一约束保障。
- 涉及关联删除的业务关系，应明确外键约束、级联策略或应用层删除策略。
- 修改表结构后，必须同步检查相关模型、业务逻辑、接口文档、初始化数据和测试用例。
- 数据库规则变更后，必须补充或更新覆盖表结构约束的自动化测试。
